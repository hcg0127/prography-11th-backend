# 🤖 AI 활용 사례 (AI Utilization Case)

본 프로젝트 진행 과정에서 LLM(Large Language Model)을 단순한 코드 생성 도구가 아닌, **페어 프로그래밍 파트너**이자 **기술 리뷰어**로 활용하여 개발 생산성과 결과물의 품질을 높였습니다.

주요 활용 분야는 **테스트 코드 검증**, **시스템 아키텍처 고도화**, **데이터 모델링 검수**, 그리고 **문서화**입니다.

---

## 1. 테스트 코드 검증 및 모킹(Mocking) 전략 수립

단위 테스트 작성 시 발생하는 Mockito 설정 오류와 스터빙(Stubbing) 불일치 문제를 해결하는 데 AI를 활용했습니다.

* **활용 상황**: `MemberQueryService` 단위 테스트 중, 서비스 로직의 변경(`findById` → `findTopBy...`)이 테스트 코드의 Mock 설정에 반영되지
  않아 `NullPointerException` 및 `AssertionError`가 발생하는 상황.
* **AI의 역할**:
    * 테스트 실패 로그 분석을 통해 "호출된 메서드"와 "Mocking된 메서드"의 불일치 포착.
    * `ArgumentMatchers.any()` 및 `anyLong()`을 활용한 유연한 스터빙 전략 제안.
    * 단순한 정답 제시가 아니라, **Strict Stubbing** 규칙과 **Argument Matcher**의 원리를 설명하여 테스트 코드의 견고함을 높임.

## 2. 시스템 아키텍처 리뷰 및 비용 최적화

초기 설계한 AWS 아키텍처에 대해 **비용 효율성**과 **가용성** 관점에서 피드백을 받고 설계를 개선했습니다.

* **활용 상황**: H2(In-Memory)를 사용하는 과제 제약 사항과 실제 운영 환경(Production)의 이상적인 아키텍처 사이의 간극을 좁히기 위한 설계 고민.
* **AI의 역할**:
    * **비용 절감 제안**: Multi-AZ 구성의 장점을 살리되, NAT Gateway 대신 NAT Instance 사용 가능성 등 비용 집중형 아키텍처에 대한 구체적 조언 제공.
    * **논리적 근거 마련**: 과제에서는 H2를 쓰지만 설계도에는 RDS를 포함한 이유를 "운영 환경을 고려한 설계(Production Readiness)"라는 논리로 정리하여 문서화 지원.

## 3. 데이터 모델링(ERD) 정규화 검수

초기 데이터 모델링 단계에서 발생할 수 있는 확장성 문제를 사전에 진단하고 정규화를 진행했습니다.

* **활용 상황**: 기수(Cohort)별로 파트(Part)와 팀(Team) 정보를 관리해야 하는데, 이를 단순 문자열로 관리할지 테이블로 분리할지 고민.
* **AI의 역할**:
    * 데이터 무결성과 향후 확장성을 고려하여 `parts`, `teams` 테이블 분리(정규화) 제안.
    * `cohort_members` 연결 테이블을 통해 회원과 조직 간의 N:M 관계를 해소하고, JPA 연관관계 매핑(`FetchType.LAZY`, `CascadeType`) 전략을 구체화.

## 4. 기술 문서 작성 및 포맷팅 (Documentation)

로직 구현에 집중할 수 있도록, 가독성 높은 문서 템플릿과 다이어그램 설명을 작성하는 데 활용했습니다.

* **활용 상황**: 방대한 설정 과정과 아키텍처 설명을 `README.md`와 별도 문서로 정리해야 하는 상황.
* **AI의 역할**:
    * **가독성 개선**: Markdown 문법(콜아웃, 테이블, 코드 블록)을 활용하여 문서의 가독성을 극대화.
    * **정보 구조화**: 복잡한 실행 방법과 환경 설정을 단계별(Step-by-Step)로 구조화하고, 오타(Mac/Linux 명령어 등)를 교정.
